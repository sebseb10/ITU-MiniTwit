\sloppy \section{System's Perspective}
\byline{Sebastian Holt Svendsen}

\noindent In this section, we provide an overview of the ITU-MiniTwit system from a technical perspective. 
We cover three areas: the architecture and design (\cref{sec:architecture}), the dependencies and technologies used (\cref{sec:dependencies}), 
and the current state of the system (\cref{sec:current_state}).
The app is deployed on DigitalOcean as a Docker Swarm cluster, fronted by two nginx load balancers,
with a shared Redis for user-session-state and a managed PostgreSQL database for persistent data.

\subsection{Architecture and Design}
\label{sec:architecture}

% Include a diagram if possible (e.g. \includegraphics{architecture.png}).
% Cover: frontend, backend, database, containerisation, orchestration.
\noindent \Cref{fig:deployment} shows the production topology
of ITU-MiniTwit deployed on DigitalOcean.
\begin{figure}[H]
    \centering
    \includegraphics[width=\textwidth]{Deployment.jpg}
    \caption{Deployment diagram --- production topology of ITU-MiniTwit deployed on DigitalOcean.}
    \label{fig:deployment}
\end{figure}

We will be going through each tier of the diagram from top-to-bottom.
\paragraph{Tier 1: Public entry.} Traffic enters via the DuckDNS domain, a free DNS 
service, resolving to a floating IP that can be reattached between
the two load balancers.

\paragraph{Tier 2: Load balancers.} Two nginx instances (LB1 MASTER, LB2 BACKUP) terminates
TLS via \texttt{Certbot} and distribute requests to the next tier. \textit{Keepalived} runs 
VRRP between them. On failue of the master, the backup promotes itself and the floating IP is 
reattached automatically.

\paragraph{Tier 3: App and data} contains two app nodes (App1, App2) each host one \texttt{itu-minitwit} replica
plus a \texttt{node-exporter}. Swarm schedules them onto nodes labelled \texttt{role == app}. 
Both replicas share a managed PostgreSQL database and a Redis instance holding session state 
and ASP.NET DataProtection keys. Because the containers hold no per-request state, replicas 
are interchangeable and a failed node does not drop sessions.

\paragraph{Tier 4: Monitoring.} A dedicated node runs Prometheus, Grafana, Loki, and Alloy. Prometheus 
scrapes \texttt{:9100} (host metrics) and \texttt{:5001} (app \texttt{/metrics}). Alloy ships container 
logs from every node to Loki. Isolation ensures an app-tier outage does not blind the diagnostic tooling.

{\sloppy \paragraph{Application architecture.} The application follows a layered design across three 
.NET projects with a one-way dependency rule. \texttt{Chirp.Core} (domain entities), \texttt{Chirp.Infrastructure} 
(EF Core \texttt{DbContext}, repositories, services, migrations), and \texttt{Chirp.Web} (Razor Pagesm scaffolded 
Identity pages, and the \texttt{MiniTwit-Stub} controller exposing the REST API). 
\\
\begin{figure}[H]
    \centering
    \includegraphics[width=\textwidth]{Sequence.png}
    \caption{Sequence diagram --- request flow to the public timeline.}
    \label{fig:sequence}
\end{figure}

\Cref{fig:sequence} taces the runtime request flow. In the common case, the browser an \texttt{HTTPS GET}: 
nginx terminates TLS and forwards over HTTP to an app replica, which reads the caller's session from Redis,
queries PostgreSQL, and returns the rendered page (\texttt{HTTP 200}). The \texttt{alt} fragment 
shows the alternative: with no valid session, the replica responds \texttt{HTTP 302 /login} and the browser 
follows the redirect. The diagram makes the responsibility split explicit, nginx only terminates TLS, 
the app is stateless, and Redis/PostgreSQL hold all shared state.

\subsection{Current State of the Systems}
\label{sec:current_state}

% Describe the current state, e.g.:
% - Results from static analysis tools (SonarQube, CodeClimate, etc.)
% - Test coverage figures
% - Known technical debt or open issues
\paragraph{Static analysis.} Code quality is tracked continuously through SonarCloud, which
analyses every push to main. The current results are shown in \cref{fig:sonarcloud}. Maintainability
is rated \textbf{A} (54 minor issues) and duplication is low at \textbf{4.7\%}, but
reliability and security both sit at \textbf{C} (20 reliability issues, 5 security issues, 
and 22 security hotspots awaiting review). The \textbf{test coverage is only 13.1\%}, the 
codebase's clearest quality weakness. Dockerfiles, shell scripts and C\# formatting are 
additionally linted in CI with hadolint, ShellCheck and \texttt{dotnet format}, configured 
to report findings without blocking the pipeline.

\begin{figure}[H]
    \centering
    \includegraphics[width=\linewidth]{SonarCube.png}
    \caption{SonarCloud analysis summary for ITU-MiniTwit.}
    \label{fig:sonarcloud}
\end{figure} 

\paragraph{What works.} In production the system is stable: the application reports healthy, 
serves the simulator's load of around \textbf{2.2k req/s} with no in-flight backlog, uses 
about \textbf{156\,MB} of memory, and returns its API endpoints in roughly \textbf{7\,ms at the 
median and 37\,ms at the 99th percentile} with no server (5xx) errors. High availability 
has been demonstrated end-to-end: keepalived failover between the two load balancers, two 
app replicas, automatic restart of crashed containers, and rolling updates without downtime
and without logging users out. The full CI/CD pipeline (build, unit and integration tests, 
end-to-end tests, static analysis, and deployment) is in place, and the metrics and logging 
stacks are deployed. Over the term the project also migrated from SQLite to a managed 
PostgreSQL database and from per-node Docker Compose to Docker Swarm.

\paragraph{Open items.} A number of limitations remain, documented here:
\begin{itemize}
    \item The public-timeline page averages about \textbf{580\,ms}, far slower than the
        API endpoints. It's the main performance weakness, rooted in the database access
        pattern and round-trips to the managed database.
    \item The logging dashboard is dominated by Loki's own internal push errors, so
        application logs are not effectively surfaced, and there is no alerting.
    \item SonarCloud's security findings are unaddressed. The Docker images are not
        security-hardened and CI has no dedicated security scanner (e.g. CodeQL or
        Trivy).
    \item There is no declarative infrastructure-as-code or configuration management
        (Vagrant, Ansible). Provisioning is handled by shell scripts, and the
        \texttt{Vagrantfile} is a non-functional stub.
    \item With only two of five group members actively contributing, some weekly tasks
        (mapping the Three Ways, security scanning, infrastructure-as-code) were not
        reached.
\end{itemize}
