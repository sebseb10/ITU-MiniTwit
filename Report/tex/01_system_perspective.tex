\sloppy \section{System's Perspective}
\byline{Sebastian Holt Svendsen}

\noindent In this section, we provide an overview of the ITU-MiniTwit system. 
We cover three areas: the architecture and design (\cref{sec:architecture}), the dependencies and technologies used (\cref{sec:dependencies}), 
and the current state of the system (\cref{sec:current_state}).
The app is deployed on DigitalOcean as a Docker Swarm cluster, fronted by two nginx load balancers,
with a shared Redis for session-state and a managed PostgreSQL database for persistent data.

\subsection{Architecture and Design}
\label{sec:architecture}

\noindent \Cref{fig:deployment} shows the production topology
of ITU-MiniTwit deployed on DigitalOcean.
\begin{figure}[H]
    \centering
    \includegraphics[width=\textwidth]{Deployment.jpg}
    \caption{Deployment diagram --- production topology of ITU-MiniTwit deployed on DigitalOcean.}
    \label{fig:deployment}
\end{figure}

We will be going through each tier of the diagram from top-to-bottom.
\paragraph{Tier 1: Public entry.} Traffic enters via the DuckDNS domain, a free dynamic DNS 
service providing public hostname. Which resolves to a floating IP that can be reattached between
the two load balancers.

\paragraph{Tier 2: Load balancers} are two nginx
instances, LB1 and LB2,
that distribute incoming requests to the webapp instances in the next tier.
\textit{Certbot} is used to automatically obtain and renew TLS
certificates for the domain. \textit{Keepalived} implements the heartbeat
mechanism: LB1 runs as MASTER and LB2 as BACKUP under VRRP. If LB1 stops responding,
keepalived on LB2 promotes itself and the Floating IP is reattached automatically, 
so the public hostname keeps resolving to a healthy node without manual intervention.

\paragraph{Tier 3: App and data} contains two app instances, the Redis instance, and the PostgreSQL database.
The app instances, App1 and App2, are each a node hosting one \texttt{itu-minitwit} replica and a \texttt{node-exporter}.
Docker Swarm schedules the replicas onto nodes labelled \texttt{role == app}. Both replicas share
two backing stores: a managed PostgreSQL database for persistent data, and a Redis instance 
on its own node holding session-state and ASP.NET DataProtection keys. Because the app container 
hold no per-request state, replicas are interchangeable: requests are distributed freely, 
and a failed node does not drop sessions.

\paragraph{Tier 4: Monitoring} is its own device with its own Docker Swarm execution environment running Prometheus,
Grafana, Loki, Alloy, and node-exporter. Prometheus scrapes metrics from every node --- \texttt{:9100} 
for node-exporter (host metrics) and \texttt{:5001} for the app's \texttt{/metrics} endpoint (HTTP and runtime metrics).
Grafana Alloy runs on every node, reads the local docker socket, and ships container logs to Loki, which
stores and indexes them. Grafana queries Prometheus and Loki to render the monitoring and logging dashboards.
Keeping observability on its own node ensures an app-tier outage does not also blind the 
diagnostic tooling.

{\sloppy \paragraph{Application architecture.} Beyond the deployment topology, the application itself 
follows a layered design split into three .NET projects with 
a one-way dependency rule. \texttt{Chirp.Core} holds the domain entities (\texttt{Author}, \texttt{Cheep}, 
\texttt{Follows}, \texttt{Recheep}). \texttt{Chirp.Infrastructure} owns persistence, \texttt{DbContext},
repositories, services behind interfaces, and the database migrations.
\texttt{Chirp.Web} is the presentation layer: ASP.NET Core Razor Pages for the user interface, scaffolded 
ASP.NET Identity pages for authentication, and a \texttt{MiniTwit-Stub} controller exposing the REST API consumed by 
the course simulator.}
\\
\\
\noindent To complement this static view, \cref{fig:sequence} traces what happens at runtime when a user requests the public timeline.
\begin{figure}[H]
    \centering
    \includegraphics[width=\textwidth]{Sequence.png}
    \caption{Sequence diagram --- request flow to the public timeline.}
    \label{fig:sequence}
\end{figure}

\noindent The browser issues an \texttt{HTTPS GET}: the active nginx instance terminates TLS 
and forwards over HTTP to an app replicas. The replica reads the caller's session from Redis, 
queries PostgreSQL for latest cheeps, and returns the rendered page with \texttt{HTTP 200}
back through nginx. The diagram makes each tier's responsibility explicit: nginx only terminates TLS, the 
app holds no per-request state, and Redis and PostgreSQL hold all shared state.
\\

the replica responds with \texttt{HTTP 302 /login}. The browser follows the redirect, and 
the user lanmds on the login page.

\subsection{Current State of the Systems}
\label{sec:current_state}

% Describe the current state, e.g.:
% - Results from static analysis tools (SonarQube, CodeClimate, etc.)
% - Test coverage figures
% - Known technical debt or open issues
\paragraph{Static analysis.} Code quality is tracked continuously through SonarCloud, which
analyses every push to main. The current results are shown in \cref{fig:sonarcloud}. Maintainability
is rated \textbf{A} (54 minor issues) and duplication is low at \textbf{4.7\%}, but
reliability and security both sit at \textbf{C} (20 reliability issues, 5 security issues, 
and 22 security hotspots awaiting review). The \textbf{test coverage is only 13.1\%}, the 
codebase's clearest quality weakness. Dockerfiles, shell scripts and C\# formatting are 
additionally linted in CI with hadolint, ShellCheck and \texttt{dotnet format}, configured 
to report findings without blocking the pipeline.

\begin{figure}[H]
    \centering
    \includegraphics[width=\linewidth]{SonarCube.png}
    \caption{SonarCloud analysis summary for ITU-MiniTwit.}
    \label{fig:sonarcloud}
\end{figure} 

\paragraph{What works.} In production the system is stable: the application reports healthy, 
serves the simulator's load of around \textbf{2.2k req/s} with no in-flight backlog, uses 
about \textbf{156\,MB} of memory, and returns its API endpoints in roughly \textbf{7\,ms at the 
median and 37\,ms at the 99th percentile} with no server (5xx) errors. High availability 
has been demonstrated end-to-end: keepalived failover between the two load balancers, two 
app replicas, automatic restart of crashed containers, and rolling updates without downtime
and without logging users out. The full CI/CD pipeline (build, unit and integration tests, 
end-to-end tests, static analysis, and deployment) is in place, and the metrics and logging 
stacks are deployed. Over the term the project also migrated from SQLite to a managed 
PostgreSQL database and from per-node Docker Compose to Docker Swarm.

\paragraph{Open items.} A number of limitations remain, documented here:
\begin{itemize}
    \item The public-timeline page averages about \textbf{580\,ms}, far slower than the
        API endpoints. It's the main performance weakness, rooted in the database access
        pattern and round-trips to the managed database.
    \item The logging dashboard is dominated by Loki's own internal push errors, so
        application logs are not effectively surfaced, and there is no alerting.
    \item SonarCloud's security findings are unaddressed. The Docker images are not
        security-hardened and CI has no dedicated security scanner (e.g. CodeQL or
        Trivy).
    \item There is no declarative infrastructure-as-code or configuration management
        (Vagrant, Ansible). Provisioning is handled by shell scripts, and the
        \texttt{Vagrantfile} is a non-functional stub.
    \item With only two of five group members actively contributing, some weekly tasks
        (mapping the Three Ways, security scanning, infrastructure-as-code) were not
        reached.
\end{itemize}
\noindent These would be the priorities for further work.
