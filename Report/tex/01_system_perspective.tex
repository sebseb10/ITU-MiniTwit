\section{System's Perspective}
\byline{Sebastian Holt Svendsen}

\noindent In this section, we provide an overview of the ITU-MiniTwit system from a technical perspective. 
We cover three areas: the architecture and design (\cref{sec:architecture}), the dependencies and technologies used (\cref{sec:dependencies}), 
and the current state of the system (\cref{sec:current_state}).
The app is deployed on DigitalOcean as a Docker Swarm cluster, fronted by two nginx load balancers,
with a shared Redis for user-session-state and a managed PostgreSQL database for persistent data.

\label{sec:architecture}

of ITU-MiniTwit deployed on DigitalOcean.
\begin{figure}[H]

We will be going through each tier of the diagram from top-to-bottom.
\paragraph{Tier 1: Public entry.} Traffic enters from the internet via the DuckDNS domain.
DuckDNS is a free dynamic DNS service for setting up a domain for ITU-MiniTwit.
The domain points to a floating IP address, which is a public IP that can be moved between different servers in the cluster, 
specifically the two load balancers.

\paragraph{Tier 2: Load balancers} are two nginx
instances, LB1 and LB2,
that distribute incoming requests to the webapp instances in the next tier. Two are deployed for higher availability.
\textit{Certbot} is used to automatically obtain and renew TLS
certificates for the domain, ensuring secure
HTTPS connections. \textit{Keepalived} implements the heartbeat
mechanism: LB1 runs as MASTER and LB2 as BACKUP under VRRP. If LB1 stops responding,
keepalived on LB2 promotes itself and the Floating IP is reattached automatically, 
so the public hostname keeps resolving to a healthy node without manual intervention.

\paragraph{Tier 3: App and data} contains two app instances, the Redis instance, and the PostgreSQL database.
The app instances, App1 and App2, are each a device running a Docker Swarm execution environment 
that hosts one \texttt{itu-minitwit} replica and a \texttt{node-exporter}.
Docker Swarm schedules the replicas onto nodes labelled \texttt{role == app}. Both replicas share
two backing stores: a managed PostgreSQL database for persistent data, and a Redis instance 
on its own node holding session state and ASP.NET DataProtection keys. Because no per-request state
is kept inside the app containers, any replica can serve any user, so the load balancers can 
distribute requests freely and fail a node out without dropping sessions. Rolling updates similarly bring
up a new container before stopping the old one, so no one is logged out.
Crashed containers are restarted automatically by Swarm's \texttt{on-failure} restart policy.

\paragraph{Tier 4: Monitoring} is its own device with its own Docker Swarm execution environment running Prometheus,
Grafana, Loki, Alloy, and node-exporter. Prometheus scrapes metrics from every node --- \texttt{:9100} 
for node-exporter (host metrics) and \texttt{:5001} for the app's \texttt{/metrics} endpoint (HTTP and runtime metrics).
Grafana Alloy runs on every node, reads the local docker socket, and ships container logs to Loki, which
stores and indexes them. Grafana queries Prometheus and Loki to render the monitoring and logging dashboards.
Keeping the observability stack on its own node means an outage of the app tier does not also bring down monitoring,
which would be needed to diagnose it. 

{\sloppy \paragraph{Application architecture.} Beyond the deployment topology, the application itself 
follows a layered design split into three .NET projects with 
a one-way dependency rule. \texttt{Chirp.Core} holds the domain entities (\texttt{Author}, \texttt{Cheep}, 
\texttt{Follows}, \texttt{Recheep}). \texttt{Chirp.Infrastructure} owns persistence, the EF Core \texttt{DbContext},
repositories, and services behind interfaces (\texttt{ICheepRepository}, \texttt{ICheepService}, etc.), and the database
migrations. \texttt{Chirp.Web} is the presentation layer: ASP.NET Core Razor Pages for the user interface, scaffolded 
ASP.NET Identity pages for authentication, and a \texttt{MiniTwit-Stub} controller exposing the REST API consumed by 
the course simulator.}

\noindent To complement this static view, \cref{fig:sequence} traces what happens at runtime when a user requests the public timeline.
\begin{figure}[H]
    \centering
    \includegraphics[width=\textwidth]{Sequence.png}
    \caption{Sequence diagram --- request flow to the public timeline.}
    \label{fig:sequence}
\end{figure}

\noindent In the common case the browser issues an \texttt{HTTPS GET}, which the active nginx instance terminates
and forwards over HTTP to one of the app replicas. The replica reads the caller's session from Redis. With a 
valid session it queries PostgreSQL for the latest cheeps and returns the rendered timeline page with \texttt{HTTP 200}
back through nginx to the browser. The diagram makes each tier's responsibility explicit: nginx only terminates TLS, the 
app holds no per-request state, and Redis and PostgreSQL hold all shared state.
\\
\noindent The \texttt{alt} fragment shows the alternative flow: when no valid session exists, the replica responds
with \texttt{HTTP 302 /login}. The browser follows this redirect transparently and re-enters the same flow against
\texttt{/login}, so the user simply lands on the login page.
